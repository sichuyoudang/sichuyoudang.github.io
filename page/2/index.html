定点数和浮点数

Posted on 2020-09-14

定点格式

小数可以表示为二进制数吗，可以，最简单的方法可能是使用 BCD码(二进制编码的十进制数)，如下：

十进制数字 BCD 代码
0 0000
1 0001
…
9 1001

通常把两个 BCD 数字存放在一个字节，这种方式成为压缩 BCD。由于2的补数不和 BCD 数放一起，所以压缩 BCD 通常需要增加1位来标识数的正负，改位成为符号位。

假设计算机程序要处理的钱款数目在 +/-100万美元之间，即范围是-9,999,999.99～9,999,999.99，因此保存的每一笔金额都需要5个字节，例如-4，325，120.25 表示为 00010100 00110010 01010001 00100000 00100101，最左边半个字节所构成的1指名该数位负数。

如果要表示的数范围扩大到-99,999,999.99～99,999,999.99，那就需要用6个字节实现，5个字节表示10个数字，另一个字节整个用来做符号位。

这种基于二进制的存储和标记方式也被称为定点格式，所谓的”定点”是指小数点的位置总是在数的某个特定位置，上例中它位于两位小数之前。需要注意的是，有关小数点位置的计数信息并没有和整个数字一起存储，所以使用定点小数的程序必须知道小数点的位置。

如果数字不大到超过预定的存储空间，并且小数位不会很多，使用定点格式的小数是一个很好的选择。但在表示非常大/小的数时就不合适。例如地球和太阳的距离490，000，000，000 英尺就需要12字节。

浮点格式

在计算机中小数的存储方式，除了定点格式还还有一种选择，即浮点格式。因为浮点格式是基于科学记数法(把每个数表示成有效位与10点幂的乘积的形式，这样可以避免写一长串0，因此这种计数方式特别适合极大或极小的数)的，但计算机中的浮点格式是借助二进制数实现的科学计数法形式，因此我们首先了解如何用二进制表示小数。

在十进制数中，小数点右边的数字与10点负整数次幂相关联，而在二进制数中，二进制小数点右边的数字和2的负整数次幂相关，如 101.1101 转化成十进制 1*4+0*2+1*1+1/2+1/4+0/8+1/16=5.8125。

在十进制的科学记数法中，规范化的有效数应该大于或等于1且小于10的，类似的在二进制的科学记数法中， 规范化的有效数应该大于或等于1且小于10的(即十进制的2)，因此在二进制的科学记数法中，101.1101 规范化应为 1.011101*22，这个规则暗示了这样一个有趣的现象，在规范化二进制浮点数中，小数点左边通常只有一个1，除此之外没有其他数字。

当代大部分计算机和计算机程序在处理浮点数时所遵循的标准是由 IEEE 于1985年制定的，ANSI 也认可，ANSI/IEEE Std754-1985称作 IEEE 二进制浮点数算数运算标准，IEEE 浮点标准定义了两种基本格式，以4个字节表示的单精度格式和以8个字节表示的双精度格式。

单精度格式

它的4个字节可以分成三部分，1位的符号位（0代表正数，1代表负数），8位用作指数位，最后的23位用作有效位：

s=1位符号 e=8位指数 f=23位有效位

3部分共32位，也就是4个字节，对于二进制科学记数法的规范，其有效位的小数点左边有且仅有一个1，因此在 IEEE 浮点数标准中，这一位没有分配存储空间。在该标准中，仅存储有效位的23位小数部分，尽管存储的只有23位，但仍然称其精度为24。

8位指数部分的取值范围是0～255，成为偏移指数，意思是对于有符号指数，为了确定其实际所代表的值必须从指数中减去一个值—称做偏移量。对于单精度浮点数，其偏移量为127。

指数0和255用于特使目的，如果指数的取值范围是1～254，那么对于一个特定的数，可以用 s(符号位), e(指数), f(有效数)来描述它: (-1)s * 1.f * 2e-127

若 s=0，则该数为正，若 s=1，该值为负。
中间 1.f 含义是 1 的后面的是小数点，小数点跟着23位有效位。1.f 和 2 的幂相乘，其中指数等于内存中的8位偏移指数减127。
如果 e=0 且 f=0，则该数为0。若 f=1， 这种情况可以解释为负0，可以用来表示极小的数，小到不能在单精度格式用数字和指数表示
如果 e=0 且 f≠0，则该数合法但不规范，该类数表示为 (-1)s * 0.f * 2-127
如果 e=0 且 f=0，则该数被解释为无穷大或无穷小，取决于 s 的值
如果 e=0 且 f≠0，则该数被解释为不是一个数，通常表示为 NaN

单精度浮点格式下，可以表示的规格化的最小正/负二进制数为 1.000000000000000000000002 * 2-126，最大正/负二进制数为 1.111111111111111111111112 * 2127，在十进制下，这两个数近似地等于 1.175494351 * 10-38 和 3.402823466 * 10-126，这也就是单精度浮点的有效表示范围。

如上，10位二进制数可以近似地用3位十进制数来表示，这就意味着：单精度浮点数格式存放的24位二进制数大体上与7位十进制数相等，所以单精度浮点格式提供24位的二进制精度或7位的十进制精度。

单精度浮点数的精度为1/224，或1/16777216，这意味什么？

这意味16777216 和 16777217 将表示成同一个数，而且处于这两个数之间的所有数（如16777216.5）也都被表示成同一个数。以同一个浮点数来表示两个不同的十进制数，有时可能成为一个问题，也可能不会。但如果银行用单精度浮点数存放带小数的数字时，就会发现262144.00和262144.01在计算机中存储为同一个数，1。这也就是为什么人们更愿意用定点数来表示钱款的原因。此外还有一个问题，你的程序经过一系列计算应得结果3，50时，由于使用浮点，可能得到的是3.49999999999，这种问题经常发生，并且没有完整解决方法。

双精度格式

如果在程序中使用浮点格式数，但使用单精度格式又出现各种问题时可以考虑双精度浮点数。这种类型需要8个字节来表示，

s=1位符号 e=11位指数 f=52位有效位

双精度浮点数的指数偏移量是1023，因此以该格式存储的数可表示为：(-1)s * 1.f * 2e-1023。上面提到的关于单精度浮点格式下的0，无穷大/小，NaN同样适用于双精度浮点格式。

双精度浮点格式下可以表示的最小正/负是：
1.00000000000000000000000000000000000000000000000000002 * 2-1022，小数点后面52个0，同样，可以表示的最大数为：1.11111111111111111111111111111111111111111111111111112 * 21023，十进制表示为2.2250738585072014* 10-308 ～ 1.7976931348623158* 10308 ，10的308次幂是一个非常巨大的数。

Python 装饰器

Posted on 2020-09-13

有时候我们不想修改函数，但是需要添加额外处理（例如记录日志，计时统计，性能监控等）时，可以在函数加上一层包装（wrapper layer）来实现。

简单装饰器函数

装饰器函数内部创建一个新函数，利用 *args 和 **kwargs来接受任意参数，在这个函数内部，调用原来的输入函数（即被包装的函数）并返回它的结果。这个新创建的 wrapper 函数作为装饰器的结果返回，取代了原来的函数。

装饰器一般不会修改调用签名，也不会修改被包装函数返回的结果，这是如何实现？

最好的方式就是使用 @wraps 装饰器，因为 @wraps 装饰器的一个重要特征就是它可以通过 __wrapped__ 属性来访问被包装的那个函数，底层的函数签名可以通过访问__wrapped__ 属性来传递。

当时要注意，不是所有的装饰器都使用了 @wraps，因此有些装饰器的行为可能和我们期望的有所区别，特殊时，由内建的装饰器 @staticmethod 和 @classmethod 构建的描述符对象并不遵循这个约定（相反，它们会把原始函数保存在__func__ 属性中），所以具体问题具体分析。

可接受参数的装饰器

编写一个函数为添加日志功能的装饰器，但是油允许用户指定日志的等级以及其他的细节作为参数。

如上，最外层的 logged() 函数接受所需的参数，并让它们对装饰器的内层函数可见，内层的 decorate() 函数接受一个函数并给它加上一个包装层，关键部分在于这个包装层可以使用传递给 logged() 的参数。
此外，decorate() 的结果必须是一个可调用对象，这个对象反过来接受一个函数作为输入，并对其进行包装。

InnoDB 两次写

Posted on 2020-09-09

如果说 Insert Buffer 带给 InnoDB 存储引擎的是性能上的提升，那个 doublewrite（两次写）带给
InnoDB 存储引擎的则是数据页的可靠性。

党发生数据库宕机时，可能 InnoDB 存储引擎正在写入某个列表中，而这个页只写了一部分，比如16KB 的页，只写了前 4KB，之后发生了宕机，这种情况称为局部写失效，在 InnoDB 存储引擎未使用 doublewrite 技术前，曾经出现过部分因为写失效而导致数据丢失的情况。

doublewrite 由两部分组成，一部分是内存中的 doublewrite buffer ，大小为 2MB，另一部分是物理磁盘上共享表空间中连续的128个页，即2个区，大小同样为2MB。在对缓冲池的脏页进行刷新时，并不直接写磁盘，而是会通过 memcpy 函数将脏页先复制到内存中的 doublewrite buffer，之后通过 doublewrite buffer 再分两次，每次1MB顺序地写入共享表空间的物理磁盘上，然后马上调用 fsync 函数，同步磁盘，避免缓冲写带来的问题。在这个过程中，因为 doublewrite buffer 页是连续的，因此这个过程是顺序写的，开销并不大，在完成 doublewrite 页的写入后，再将 doublewrite buffer 中的页写入各个表空间文件中，此时的写入则是离散的。

如果操作系统在将页写入磁盘的过程中发生了崩溃，在恢复过程中，InnoDB 存储引擎可以从共享表空间中的 doublewrite 中找到该页的一个副本，将其复制到表空间文件，再应用重做日志。

MySQL 官方手册中，命令 SHOW GLOBAL STATUS 中 buffer_pool_pages_flushed 变量表示当前从缓冲池中刷新首先都需要放入到 doublewrite 中，因此该变量应该和 InnoDB_dblwr_pages_written 一致，然而在 MySQL 5.5.24 版本之前，buffer_pool_pages_flushed 总是 InnoDB_dblwr_pages_written 的2倍，直至5.5.24才修复。因此用户若需要统计数据库在生产环境中写入到量，最安全的还是根据 InnoDB_dblwr_pages_written 来统计，这在所有版本中都是正确的。

参数 skip_innodb_doublewrite 可以禁止 doublewrite 功能，这时可能会发生前面提及的写失效问题。不过如果用户有多个从服务器，需要提供较快的性能，也行启用这个参数是一个办法。不过对于需要提供数据高可靠性的主服务器，任何时候用户都应确保开启 doublewrite 功能。

InnoDB 逻辑存储结构

Posted on 2020-09-09

从 InnoDB 存储引擎的逻辑存储结构看，所有数据都被逻辑的存放在一个空间中，成为表空间（tablespace），表空间又由段（segment），区（extent），页（page）组成，页在一些文档中有时也称为块（block）。

表空间

表空间可以看作是 InnoDB 存储引擎逻辑结构的最高层，所有数据都存放在表空间中，默认情况下 InnoDB 存储引擎有一个共享空间 ibdata1，即所有数据都存放在这个表空间内，如果用户启用了参数 innodb_file_per_table,则每张表内的数据可以单独放到一个表空间内。

如果启用了 innodb_file_per_table 参数，每张表的表空间内存放的只是数据，索引和插入缓冲 Bitmap 页，其他类的数据如回滚信息，插入缓冲索引页，系统事务信息，二次写等还是放在原来的共享表空间内。因此技术启用了参数 innodb_file_per_table ，共享表空间还是会不断增加其大小。

段

常见的段有数据段，索引段，回滚段等，因为 InnoDB 存储引擎表是索引组织的，因此数据即索引，索引即数据。那么数据段即为 B+ 树的叶子节点（Leaf node segment），索引段即为 B+ 树的非索引节点（Non-leaf node segment），回滚段较为特殊。

在 InnoDB 存储引擎中，对段段管理都是由引擎自身完成。DBA 不能也没必要对其进行控制。

区

区是由连续页组成的空间，在任何情况下每个区的大小都是 1MB。为了保证区中页的连续性，InnoDB 存储引擎一次从磁盘申请 4～5 个区。在默认情况下，InnoDB 存储引擎页的大小为 16KB，即一个区中一共有64 个连续的页。

InnoDB 1.0.x 版本开始引入压缩页，即每个页的大小可以通过参数 KEY_BLOCK_SIZE 设置为2K，4K，8K，因此每个区对应页的数量就应该为512，256，128。

InnoDB 1.2.x 版本新增了参数 innodb_page_size，通过该参数可以将默认页的大小设置为4K，8K，但是页中的数据库不是压缩。这时区中的数量同样也为256，128。总之不论页的大小怎么变化，区的大小总是1MB。

但是还有一个问题，在用户启用了参数 innodb_file_per_table 后，创建的表默认大小是96KB，区中是64个连续的页，创建的表的大小至少是 1MB 才对啊？其实这是因为在每个段开始时，先用32个页大小的碎片页来存放数据，在使用完这些页之后才是64个连续页的申请。这样做的目的是，对于一些小表，或者是 undo 这类的段，可以在开始时申请较少的空间，节省磁盘容量的开销。

页

同大多数数据库一样，InnoDB 有页的概念（也可称为块），页是 InnoDB 磁盘管理的最小单位。在 InnoDB 存储引擎中，默认每个页的大小为 16KB。而从 InnoDB 1.2.x 版本开始，可以通过参数 innodb_page_size 将页的大小设置为 4K, 8K, 16k。若设置完成，则表中页的大小都为 innodb_page_size，不可以对其再次进行修改，除非通过 mysqldump 导入和导出操作来产生新的库。

在 nnoDB 存储引擎中，常见的页类型有：

数据页 （B-tree Node）
undo 页（undo Log Page）
系统页（System Page）
事务数据页（Transaction system Page）
插入缓冲位图页（Insert Buffer Bitmap）
插入缓冲空闲列表页（Insert Buffer Free List）
未压缩的二进制大对象页（Uncompressed BLOB Page）

Mysql 结构体系

MySQL Server 体系结构

Connectors(连接者)：指的是不同语言和SQL 的交互，如上图可知目前流行的语言都支持 MySQL 客户端连接

Connection Pool(连接池)：管理缓冲用户连接，线程处理等需要缓存的需求，在这里也会进行用户账号密码和库表权限验证

SQL Interface(SQL接口)：接受用户执行的SQL 语句，并返回查询的结果

Parser(查询解析器)：SQL命令传递到解析器的时候会被解析器验证和解析(权限、语法结构)

Optimizer(查询优化器)：SQL语句在查询之前会使用查询优化器对查询进行优化

Cache $ Buffer (缓存&缓冲)：主要包含 QC（查询缓存）以及表缓存，权限缓存等，QC 以往主要用于 MyISAM 存储引擎，目前在 MySQL8.0 中已放弃，对于现在非常流行的 InnoDB 存储引擎来说没有意义，因为InnoDB 存储引擎 有自己非常完善的缓存功能。

Pluggable Storage Engines(插件式存储引擎): 存储引擎是 MySQL 中具体和文件打交道的子系统，

Files & Logs (磁盘物理文件) ： 包含 MySQL 各个引擎的数据，索引的文件，以及各种日志文件。

File System (文件系统)：对存储设备的空间进行组织和分配，负责文件存储并对存入的文件进行保护和检索的系统，他负责为用户建立文件，存入，读出，修改，转储文件，控制文件的存取。

MySQL 中一个查询会话请求流程

  1. MySQL 客户端对 MySQL server 的监听端口发起请求
  2. 在连接者组建层创建连接，分配线程，并验证用户名，密码和库表权限
  3. 如果打开了 query_cache，则检查之，有数据直接返回，没有继续往下执行
  4. SQL 接口组建接受 SQL 语句，将 SQL 语句分解成数据结构，并将这个结构传递到后续步骤中（将 SQL 语句解析成 MySQL 认识的语法）
  5. 查询优化器组建生成查询路径树，并选举一条最优的查询路径
  6. 调用存储引擎接口，打开表，执行查询，检查存储引擎缓存中是否有对应的缓存记录，如果没有就继续往下执行
  7. 到磁盘物理文件中寻找数据
  8. 到查询到所需要的数据之后，先卸乳存储引擎缓存中，如果打开了query_cache，也会同时写进去
  9. 返回数据给客户端
  10. 关闭表
  11. 关闭线程
  12. 关闭连接

线程安全

在并发编程时，如果多个线程访问同一资源，我们需要保证访问的时候不会产生冲突，数据修改不会发生错误，这就是我们常说的 线程安全 。

一个 Flask 应用启动之后，请求发来的时候执行了app（Flask的实例化对象）的__call__方法，而__call__方法返回了app 的 wsgi_app(environ, start_response)方法的执行结果:

每一个请求到来之后，Flask 会为它新建一个 request_context，这个对象的上下文包含着 request 相关的信息：

Flask 创建RequestContext 对象后，将这个对象 push 进全局变量 _request_ctx_stack中，在 push 前还要检查_app_ctx_stack，如果_app_ctx_stack的栈顶元素不存在或是与当前的应用不一致，则首先push appcontext 到_app_ctx_stack中，再push requestcontext。

通过上面的两步，每一个请求的应用上下文和请求上下文就被 push 到了全局变量_request_ctx_stack和_app_ctx_stack中。

现在知道了_request_ctx_stack和_app_ctx_stack是何时被push的，每一个请求到来都会导致新的 RequestContext和AppContext被建立并push，一旦请求处理完毕就被pop出去。而无论是_app_ctx_stack还是_request_ctx_stack都是LocalStack对象，这是 werkzeug 中的一个对象，这个对象的几乎所有重要属性在_local这一属性中，它是一个 Local对象:
