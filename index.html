Vaex 使用

Posted on 2021-07-08

vaex 定义

Pandas 把数据读取到内存中，然后进行处理和计算。而 Vaex 只会对数据进行内存映射，而不是真的读取数据到内存中，这个和 Spark 的懒加载是一样的，在使用的时候 才会去加载，声明的时候不加载。

所以说不管加载多大的数据，10GB、100GB…对 Vaex 来说都是瞬间搞定。不过，Vaex 的懒加载只支持 HDF5, Apache Arrow,Parquet, FITS 等文件，不支持 csv 等文本文件，因为文本文件没办法进行内存映射。

Vaex 全过程都是零内存，它的处理过程仅仅产生 expression（表达式），表达式是逻辑表示，不会执行，只有到最后的生成结果阶段才会执行，整个过程数据是流式传输，不产生内存积压。

Ray 使用经验

Posted on 2021-06-17

定义

Ray 是一个分布式执行引擎，多 worker 进程执行多个任务并将结果存储在对象存储中，每个 worker 都是一个独立的进程。

每个节点上的对象存储都将不可变的对象存储存储在共享内存中，允许 worker 以少量的复制和并行化有效的共享同一节点上的存储对象。远程对象不可变，它们的值在创建之后不能更改，这样只需在多个对象存储中复制远程对象，而不需要同步副本。

ray.get 和 ray.put 是用作 python 对象和对象 ID 之间的转换。

示例

进程存储和共享存储

生成

  1. 由 worker 进程通过 ray.get 或 ray.wait 请求的。一旦 worker 完成 ray.get 请求，并且Python 对对象值的引用超出范围，就可以释放这些对象
  2. 由在该节点上执行的任务返回，一旦不再有对对象的引用或者对象被溢出，就可以释放这些引用
  3. 由该节点上的 worker 进程通过 ray.put 创建的。一旦不再引用对象，就可以释放这些对象
  4. 该节点上排队的任务的参数，一旦任务完成或不再排队，就可以释放它们
  5. 以前这个节点上需要的

内存分配

内存分配从低到高，内存释放从 6 开始，直至有足够计算的内存：

  1. worker 中使用的对象（ ray.get 请求或运行 task 的参数）（任务完成之前不能释放参数。默认上限是对象存储内存的70%）
  2. 创建对象的主副本 （ray.put 生成或 task 返回）
  3. ray.get 或 ray.wait 请求参数
  4. 队列任务的参数
  5. 以前使用的对象
  6. 未分配内存

主拷贝和可回收拷贝

  1. 主副本是唯一的，因为只要所有者对对象的引用计数大于0，它就不会被释放。对象的其他副本可能在本地内存压力下被 LRU 逐出。
  2. 在大多数情况下，主副本是要创建的对象的第一个副本。如果初始副本因故障而丢失，所有者将尝试根据对象的可用位置指定新的主副本。
  3. 一旦对象引用计数变为0，对象的所有副本最终都会被自动垃圾收集。所有者会立即从进程内存储中删除小对象。raylet 从分布式对象存储中异步擦除大型对象。
  4. raylet 还管理分布式对象传输，它根据当前需要对象的位置创建对象的附加副本，例如，如果依赖对象的任务被调度到远程节点。

对象溢出和持久性

Ray 1.3.0+默认支持在对象存储的容量用完后将对象溢出到外部存储。这使得核心外数据处理和内存密集型分布式应用程序成为可能。

两种类型的外部存储：

  1. 本地存储。默认情况下选择”本地磁盘”，以便 Ray 用户可以使用对象溢出功能，而无需任何其他配置。
  2. 分布式存储（S3）。访问速度可能较慢，但这可以提供更好的容错性，因为数据将在 worker 节点故障后继续存在。

调度

Ray使用了混合任务调度的思路。在集群部署模式下，Ray 启动了以下关键组件：

GlobalScheduler：Master 上启动了一个全局调度器，用于接收本地调度器提交的任务，并将任务分发给合适的本地任务调度器执行

RedisServer：Master 上启动了一到多个 RedisServer 用于保存分布式任务的状态信息（ControlState），包括对象机器的映射、任务描述、任务 debug 信息等。

LocalScheduler：每个 Slave 上启动了一个本地调度器，用于提交任务到全局调度器，以及分配任务给当前机器的 Worker 进程

Worker：每个 Slave 上可以启动多个 Worker 进程执行分布式任务，并将计算结果存储到ObjectStore

ObjectStore：每个 Slave 上启动了一个 ObjectStore 存储只读数据对象，Worker 可以通过共享内存的方式访问这些对象数据，这样可以有效地减少内存拷贝和对象序列化成本。ObjectStore 底层由Apache Arrow 实现

Plasma：每个 Slave 上的 ObjectStore 都由一个名为 Plasma 的对象管理器进行管理，它可以在 Worker 访问本地 ObjectStore 上不存在的远程数据对象时，主动拉取其它 Slave 上的对象数据到当前机器

celery_use

Posted on 2021-06-02

Celery 管理分布式消息队列的工具，它的主要工作是管理分配任务到不同的服务器，并取得结果。

执行任务时需要一个消息中间件接受和发送任务消息和存储任务结果，RabbitMQ 和 Redis 中间人的消息传输支持所有特性，但也提供大量其他实验性方案的支持。

Celery 有两大使用场景：

异步： 用户生成任务，任务执行时间太长，可以放在 Celery 任务队列中，并获取其 task_id，后续通过 Celery 得知任务进行状态和进度，任务完成后获取运行结果。

定时：需要定时执行的任务，也可以通过 celery beat 来实现。

角色

Producer : 任务生产者
调用 Celery API 函数或者装饰器，而产生任务并交给任务队列处理的都是任务生产者。

Celery Beat : 任务调度器
Beat 进程会读取配置文件的内容，周期性的将配置中到期需要执行的任务发送给任务队列。

Message Broker：消息中间件

当程序调用 Celery 异步任务时，接受发送过来的任务消息，存进队列而后 worker 会根据规则消费这些消息。Celery本身不提供消息服务，但是可以方便的和第三方提供的消息中间件集成。包括，RabbitMQ, Redis, MongoDB (experimental), Amazon SQS (experimental),CouchDB (experimental), SQLAlchemy (experimental),Django ORM (experimental), IronMQ 等。

Worker：消费者

执行任务的消费者，通常会在多台服务器运行多个消费者，提高运行效率，会根据配置的规则获取指定消息。

Result Backend：任务执行结果存储

任务处理完成之后保存状态信息和结果，以供查询，支持不同方式存储任务的结果，包括 AMQP, redis，memcached, mongodb，SQLAlchemy, Django ORM，Apache Cassandra, IronCache 等。

Redis 持久化

Posted on 2021-04-20

Redis 的普遍使用场景是在内存中存储数据，但如果系统宕机了，数据就会消失，为了预防这个问题，Redis 提供了两种机制： AOF(Append Only File)日志和 RDB 快照(Napshotting)。

AOF

AOF 是先执行命令，把数据写入内存，然后再记录日志。其中日志记录的是每一步是记录 Redis 收到的每一条命令，这些命令以文本形式保存。

优点

  1. 为避免额外开销，写日志时不会对命令进行语法检查，所以如果先写日志，日志中可能记录了错误的命令，Redis 使用日志恢复数据时，就可能出错。而后写日志，就是让系统先执行命令，命令执行成功再写入日志，所以 Redis 用后写日志方式可以避免出现错误命令的情况。
  2. 后写日志不会阻塞当前的写操作。

风险

  1. 刚执行完一个命令，系统宕机了，命令和数据有丢失的风险
  2. AOF 虽避免了对当前命令的阻塞，但可能会给下一个操作带来阻塞风险

上面风险都是和 AOF 写回磁盘的时机相关，所以解决了时机问题，上面的风险就大大降低了。

三种写回策略

对于上面的问题， AOF 配置项 appendfsync 有3种可选项：

Always

同步写回，每个写命令执行完，立马同步将日志写回磁盘

影响：数据基本不会丢失，但在每个写命令后有一个慢落盘的操作，影响主线程性能

Everysec

每秒写回，每个写命令执行完，先把日志写到 AOF 文件的内存缓冲区，每隔1秒把缓冲区的内容写入磁盘

影响：如果 AOF 没有写回磁盘，一旦宕机对应的数据就丢失了，丢失1秒的数据

No

操作系统控制的写回，每个写命令执行完，先把日志写到 AOF 文件的内存缓冲区，由操作系统决定何时把缓冲区的内容写入磁盘

影响：在影响主性能和数据丢失两者之间取个折中

文件过大怎么办

  1. 文件系统本身对文件大小有限制，无法保存过大的文件
  2. 文件太大的话，后续往里面追加命令的时候效率会变低
  3. 如果发生当即，AOF 里面的命令被一个个重新执行，如果文件太大，恢复过程会很慢

因此要采用一定的控制手段，例如 AOF 重写机制。

AOF 重写机制

在重写时，Redis 根据数据库现状创建一个新的 AOF 文件，即读取数据库的所有键值对，然后对每一个键值对用一条命令记录它的写入，实现 “多变一” 功能，即旧日志文件中的多条命令，在重新的新日志中变成了一条命令，因为当一个键值被多条命令反复修改时，重写时只写入键值的最新状态。

和 AOF 日志由主进程写回不同，重写过程是由后台进程 bgrewriteaof 来完成的，避免了阻塞主进程。每次执行重写时，主线程 fork 出后台的 bgrewriteaof 子进程，把主进程的内存拷贝一份给子进程，里面包含数据库的最新数据，数据写入日志时不影响主进程。

因为主进程未阻塞，仍然可以处理新来的操作，所以有写操作时，第一处日志（正在使用的 AOF 日志），Redis 会把这个操作写进它的缓冲区，这样即使宕机，这个日志的操作仍然是齐全可操作的。

第二处日志（新的 AOF 重写日志），这个操作也会被写到重写日志的缓冲区，这样重写日志也不会丢失最新的操作。等拷贝数据的所有操作记录重写完成后，重写日志记录的最新操作也会写入新的 AOF 日志，以保证数据库最新状态的记录，此时，可以用新的 AOF 文件替代旧文件了。

总之，每次 AOF 重写时， Redis 会先执行一个内存拷贝，用于重写，然后使用两个日志保证重写过程中，新写入的数据不会丢失。而且因为 Redis 采用额外线程进行重写，不会造成主线程的堵塞。

Pair RDD

提取每个元素的第一个元素作为 key 剩余元素作为 value 创建 Pair RDD，Pair RDD 由于有 key 的存在，与普通的 RDD 相比更加格式化，这种特性就会给 Pair RDD 赋予一些特殊的操作，例如 groupByKey() 可以将具有相同 key 进行分组，其结果仍然得到 Pair RDD，然后利用 mapValues() 对相同 key 的 value 进行函数计算，reduceByKey()、countByKey()和sortByKey()等一系列ByKey()操作
