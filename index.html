Redis 持久化

Posted on 2021-04-20

Redis 的普遍使用场景是在内存中存储数据，但如果系统宕机了，数据就会消失，为了预防这个问题，Redis 提供了两种机制： AOF(Append Only File)日志和 RDB 快照(Napshotting)。

AOF

AOF 是先执行命令，把数据写入内存，然后再记录日志。其中日志记录的是每一步是记录 Redis 收到的每一条命令，这些命令以文本形式保存。

优点

  1. 为避免额外开销，写日志时不会对命令进行语法检查，所以如果先写日志，日志中可能记录了错误的命令，Redis 使用日志恢复数据时，就可能出错。而后写日志，就是让系统先执行命令，命令执行成功再写入日志，所以 Redis 用后写日志方式可以避免出现错误命令的情况。
  2. 后写日志不会阻塞当前的写操作。

风险

  1. 刚执行完一个命令，系统宕机了，命令和数据有丢失的风险
  2. AOF 虽避免了对当前命令的阻塞，但可能会给下一个操作带来阻塞风险

上面风险都是和 AOF 写回磁盘的时机相关，所以解决了时机问题，上面的风险就大大降低了。

三种写回策略

对于上面的问题， AOF 配置项 appendfsync 有3种可选项：

Always

同步写回，每个写命令执行完，立马同步将日志写回磁盘

影响：数据基本不会丢失，但在每个写命令后有一个慢落盘的操作，影响主线程性能

Everysec

每秒写回，每个写命令执行完，先把日志写到 AOF 文件的内存缓冲区，每隔1秒把缓冲区的内容写入磁盘

影响：如果 AOF 没有写回磁盘，一旦宕机对应的数据就丢失了，丢失1秒的数据

No

操作系统控制的写回，每个写命令执行完，先把日志写到 AOF 文件的内存缓冲区，由操作系统决定何时把缓冲区的内容写入磁盘

影响：在影响主性能和数据丢失两者之间取个折中

文件过大怎么办

  1. 文件系统本身对文件大小有限制，无法保存过大的文件
  2. 文件太大的话，后续往里面追加命令的时候效率会变低
  3. 如果发生当即，AOF 里面的命令被一个个重新执行，如果文件太大，恢复过程会很慢

因此要采用一定的控制手段，例如 AOF 重写机制。

AOF 重写机制

在重写时，Redis 根据数据库现状创建一个新的 AOF 文件，即读取数据库的所有键值对，然后对每一个键值对用一条命令记录它的写入，实现 “多变一” 功能，即旧日志文件中的多条命令，在重新的新日志中变成了一条命令，因为当一个键值被多条命令反复修改时，重写时只写入键值的最新状态。

和 AOF 日志由主进程写回不同，重写过程是由后台进程 bgrewriteaof 来完成的，避免了阻塞主进程。每次执行重写时，主线程 fork 出后台的 bgrewriteaof 子进程，把主进程的内存拷贝一份给子进程，里面包含数据库的最新数据，数据写入日志时不影响主进程。

因为主进程未阻塞，仍然可以处理新来的操作，所以有写操作时，第一处日志（正在使用的 AOF 日志），Redis 会把这个操作写进它的缓冲区，这样即使宕机，这个日志的操作仍然是齐全可操作的。

第二处日志（新的 AOF 重写日志），这个操作也会被写到重写日志的缓冲区，这样重写日志也不会丢失最新的操作。等拷贝数据的所有操作记录重写完成后，重写日志记录的最新操作也会写入新的 AOF 日志，以保证数据库最新状态的记录，此时，可以用新的 AOF 文件替代旧文件了。

总之，每次 AOF 重写时， Redis 会先执行一个内存拷贝，用于重写，然后使用两个日志保证重写过程中，新写入的数据不会丢失。而且因为 Redis 采用额外线程进行重写，不会造成主线程的堵塞。

Python 上下文管理器

Posted on 2020-10-13

with 块

with 语句会设置一个临时的上下文，交给上下文管理器对象控制，并且负责清理上下文。这样既能避免错误又能减少样板代码。with 语句的目的是简化 try/finally 模式，这种模式用于保证一段代码运行完毕后执行某项操作，即便那段代码由于异常，return 语句或 sys.exit() 调用而终止，也会执行指定的操作。finally 字句中的代码通常用于释放重要资源，或者还原临时变更的状态。

上下文管理器协议包括 __enter__ 和 __exit__ 两个方法。 with 语句开始运行时，会在上下文管理器对象上调用 __enter__ 方法，返回除了上下文管理器之外，还可能返回其他对象，结束时调用 __exit__ 。

with 语句的 as 子句可选，对于 open 函数来说必须加上 as 子句，以便获取文件的引用。不过有些上下文管理器会返回 None，因为没什么有用的对象能提供给用户。

参上，传给 __exit__ 方法的三个参数列举如下：

exc_type 异常类
exc_value 异常实例，有时会有参数传给异常构造方法，例如错误消息，这些参数可以使用 exc_value.args 获取
traceback traceback 对象，在 try/finally 块中调用 sys.exc_info() 得到的就是 __exit__ 接收的这三个参数

contextlib 模块

contextlib 模块中有一些类和函数也能实现上下文管理器：

closing
如果对象提供了 close() 方法，但没有实现 __enter__/__exit__ 协议，那么可以使用这个函数构建上下文管理器。

supress
构建临时忽略指定异常的上下文管理器

@contextmanager
这个装饰器把简单的生成器函数变成上下文管理器，这样就不用创建类去实现管理器协议了。

ContextDecorator
基类，用于定义基于类的上下文管理器。这种上下文管理器也用于装饰函数，在受管理的上下文中运行整个函数。

ExitStack
这个上下文管理器能进入多个上下文管理器。with 块结束时，ExitStack 按照后进先出的顺序调用栈中各个上下文管理器的 __exit__。如果事先不知道 with 块要进入多少个上下文管理器，可以使用这个类，例如同时打开任意一个文件列表中的所有文件。

使用 @contextmanager

@contextmanager 装饰器能减少创建上下文管理器的样板代码，因为不用编写一个完整的类，定义 __enter__ 和 __exit__ 方法，而只需实现一个 yield 语句的生成器，生成想让 __enter__ 方法返回的值。

在使用 @contextmanager 装饰的生成器中， yield 语句的作用是把函数的定义体分成两部分，yield 语句前面的所有代码在 with 块开始时（调用 __enter__ 方法时）执行，yield 语句后面的代码在 with 块结束时（调用 __exit__ 方法时）执行。

其实 @contextmanager 装饰器会把函数包装成 __enter__ 和 __exit__ 方法的类，这个类的 __enter__ 方法有如下作用：

调用生成器函数，保存生成器对象（此处称为 gen）
调用 next(gen) ，执行到 yield 关键字所在的位置
返回 next(gen) 产出的值，以便把产出的值绑定到 with/as 语句中的目标变更上

with 块终止时，__exit__ 方法会做以下几件事：

检查有没有异常传给 exc_type，如果有，调用 gen.throw(exception)，在生成器函数定义体中包含 yield 关键字的哪一行抛出异常
否则，调用 next(gen) ，继续执行生成器函数定义体中的 yield 语句之后的代码

使用 property 需要注意事项:

如果 property 没有完成任何额外的处理任务，就不要实现这种方式，因为一是代码会变啰嗦，对其他人来说比较困惑，二是这样会让程序变慢很多。特别是如果稍后要对某个普通属性增加额外处理步骤时，可以在不修改已有代码的情况下将这个属性提升为一个 property。因为访问一个属性的语法不会变（访问普通属性和访问 property 属性的代码写法一样）。

此外不要编写那种定义了大量重复性 property 的代码，这样的代码膨胀，容易出错，这种情况下使用描述符更好。

2. 使用描述符类

所谓描述符就是以特殊方法 __get__() __set__() __delete__()的形式实现了三个核心的属性访问操作（对应 get， set， delete）的类。这些方法通过接受类实例作为输入来工作，之后底层实例字典会根据需要适当进行调整。

每个描述符方法都会接受被操纵的实例作为输入，再执行所请求的操作，底层的实例字典会调整。描述符的 self.name 属性会保存字典的健，通过这些健可以找到存储在实例字典的实际数据。对于大多数 Python 类的属性，描述符都提供了底层的魔法，包括 @classmethod, @staticmethod, @property 甚至 __solts__。

定点数和浮点数

Posted on 2020-09-14

定点格式

小数可以表示为二进制数吗，可以，最简单的方法可能是使用 BCD码(二进制编码的十进制数)，如下：

十进制数字 BCD 代码
0 0000
1 0001
…
9 1001

通常把两个 BCD 数字存放在一个字节，这种方式成为压缩 BCD。由于2的补数不和 BCD 数放一起，所以压缩 BCD 通常需要增加1位来标识数的正负，改位成为符号位。

假设计算机程序要处理的钱款数目在 +/-100万美元之间，即范围是-9,999,999.99～9,999,999.99，因此保存的每一笔金额都需要5个字节，例如-4，325，120.25 表示为 00010100 00110010 01010001 00100000 00100101，最左边半个字节所构成的1指名该数位负数。

如果要表示的数范围扩大到-99,999,999.99～99,999,999.99，那就需要用6个字节实现，5个字节表示10个数字，另一个字节整个用来做符号位。

这种基于二进制的存储和标记方式也被称为定点格式，所谓的”定点”是指小数点的位置总是在数的某个特定位置，上例中它位于两位小数之前。需要注意的是，有关小数点位置的计数信息并没有和整个数字一起存储，所以使用定点小数的程序必须知道小数点的位置。

如果数字不大到超过预定的存储空间，并且小数位不会很多，使用定点格式的小数是一个很好的选择。但在表示非常大/小的数时就不合适。例如地球和太阳的距离490，000，000，000 英尺就需要12字节。

浮点格式

在计算机中小数的存储方式，除了定点格式还还有一种选择，即浮点格式。因为浮点格式是基于科学记数法(把每个数表示成有效位与10点幂的乘积的形式，这样可以避免写一长串0，因此这种计数方式特别适合极大或极小的数)的，但计算机中的浮点格式是借助二进制数实现的科学计数法形式，因此我们首先了解如何用二进制表示小数。

在十进制数中，小数点右边的数字与10点负整数次幂相关联，而在二进制数中，二进制小数点右边的数字和2的负整数次幂相关，如 101.1101 转化成十进制 1*4+0*2+1*1+1/2+1/4+0/8+1/16=5.8125。

在十进制的科学记数法中，规范化的有效数应该大于或等于1且小于10的，类似的在二进制的科学记数法中， 规范化的有效数应该大于或等于1且小于10的(即十进制的2)，因此在二进制的科学记数法中，101.1101 规范化应为 1.011101*22，这个规则暗示了这样一个有趣的现象，在规范化二进制浮点数中，小数点左边通常只有一个1，除此之外没有其他数字。

当代大部分计算机和计算机程序在处理浮点数时所遵循的标准是由 IEEE 于1985年制定的，ANSI 也认可，ANSI/IEEE Std754-1985称作 IEEE 二进制浮点数算数运算标准，IEEE 浮点标准定义了两种基本格式，以4个字节表示的单精度格式和以8个字节表示的双精度格式。

单精度格式

它的4个字节可以分成三部分，1位的符号位（0代表正数，1代表负数），8位用作指数位，最后的23位用作有效位：

s=1位符号 e=8位指数 f=23位有效位

3部分共32位，也就是4个字节，对于二进制科学记数法的规范，其有效位的小数点左边有且仅有一个1，因此在 IEEE 浮点数标准中，这一位没有分配存储空间。在该标准中，仅存储有效位的23位小数部分，尽管存储的只有23位，但仍然称其精度为24。

8位指数部分的取值范围是0～255，成为偏移指数，意思是对于有符号指数，为了确定其实际所代表的值必须从指数中减去一个值—称做偏移量。对于单精度浮点数，其偏移量为127。

指数0和255用于特使目的，如果指数的取值范围是1～254，那么对于一个特定的数，可以用 s(符号位), e(指数), f(有效数)来描述它: (-1)s * 1.f * 2e-127

若 s=0，则该数为正，若 s=1，该值为负。
中间 1.f 含义是 1 的后面的是小数点，小数点跟着23位有效位。1.f 和 2 的幂相乘，其中指数等于内存中的8位偏移指数减127。
如果 e=0 且 f=0，则该数为0。若 f=1， 这种情况可以解释为负0，可以用来表示极小的数，小到不能在单精度格式用数字和指数表示
如果 e=0 且 f≠0，则该数合法但不规范，该类数表示为 (-1)s * 0.f * 2-127
如果 e=0 且 f=0，则该数被解释为无穷大或无穷小，取决于 s 的值
如果 e=0 且 f≠0，则该数被解释为不是一个数，通常表示为 NaN

单精度浮点格式下，可以表示的规格化的最小正/负二进制数为 1.000000000000000000000002 * 2-126，最大正/负二进制数为 1.111111111111111111111112 * 2127，在十进制下，这两个数近似地等于 1.175494351 * 10-38 和 3.402823466 * 10-126，这也就是单精度浮点的有效表示范围。

如上，10位二进制数可以近似地用3位十进制数来表示，这就意味着：单精度浮点数格式存放的24位二进制数大体上与7位十进制数相等，所以单精度浮点格式提供24位的二进制精度或7位的十进制精度。

单精度浮点数的精度为1/224，或1/16777216，这意味什么？

这意味16777216 和 16777217 将表示成同一个数，而且处于这两个数之间的所有数（如16777216.5）也都被表示成同一个数。以同一个浮点数来表示两个不同的十进制数，有时可能成为一个问题，也可能不会。但如果银行用单精度浮点数存放带小数的数字时，就会发现262144.00和262144.01在计算机中存储为同一个数，1。这也就是为什么人们更愿意用定点数来表示钱款的原因。此外还有一个问题，你的程序经过一系列计算应得结果3，50时，由于使用浮点，可能得到的是3.49999999999，这种问题经常发生，并且没有完整解决方法。

双精度格式

如果在程序中使用浮点格式数，但使用单精度格式又出现各种问题时可以考虑双精度浮点数。这种类型需要8个字节来表示，

s=1位符号 e=11位指数 f=52位有效位

双精度浮点数的指数偏移量是1023，因此以该格式存储的数可表示为：(-1)s * 1.f * 2e-1023。上面提到的关于单精度浮点格式下的0，无穷大/小，NaN同样适用于双精度浮点格式。

双精度浮点格式下可以表示的最小正/负是：
1.00000000000000000000000000000000000000000000000000002 * 2-1022，小数点后面52个0，同样，可以表示的最大数为：1.11111111111111111111111111111111111111111111111111112 * 21023，十进制表示为2.2250738585072014* 10-308 ～ 1.7976931348623158* 10308 ，10的308次幂是一个非常巨大的数。
